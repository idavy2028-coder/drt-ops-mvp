# 通渭县车辆位置管理试点运行手册

## 适用范围

本手册用于通渭县 1 个月试点期间的人工车辆位置上报与异常处置。当前模块使用内置账号和 JWT；不包含 GPS/司机端接入，也不替代真实高德地图、电子围栏、地址搜索或路径规划的独立验收。

## 本机启动顺序

1. 启动 PostgreSQL/PostGIS：`docker compose -f infra/docker-compose.yml up -d`，确认数据库健康后再启动应用。
2. 设置仅用于本机的认证占位变量并启动 API：`DRT_AUTH_JWT_SECRET`、`DRT_AUTH_BOOTSTRAP_ADMIN_USERNAME`、`DRT_AUTH_BOOTSTRAP_ADMIN_PASSWORD`。三项均不得使用真实生产密码。
3. 启动算法服务：在 `apps/algorithm` 创建虚拟环境、安装依赖后运行 Uvicorn，默认端口 `8090`。
4. 启动管理端：在 `apps/admin-web` 安装依赖并运行开发服务，默认端口 `5173`；如 API 不在同源地址，设置 `VITE_API_BASE_URL`。
5. 打开 API、算法服务和管理端健康检查，确认三个进程可用后开始值班。

位置超时阈值由 `VITE_MANUAL_LOCATION_STALE_MINUTES` 配置，未设置时页面使用 30 分钟。试点值班前应记录实际配置值和生效时间，不得在本手册中记录 JWT、初始管理员密码或其他真实凭据。

## 每班上报 SOP

1. 调度员通过电话或微信接收驾驶员反馈，保留业务沟通凭据所在的合规渠道编号。
2. 在任务位置确认面板选择当前车辆和对应任务节点，录入经纬度或已核验的虚拟站点。
3. 填写驾驶员实际反馈时间；该时间不得晚于服务器当前时间。系统自动记录调度员的录入时间，两个时间不得混用。
4. 按任务实际进展依次提交：发车、到达上车点、乘客上车、到达目的地、乘客下车、完成。
5. 提交后核对页面的最新位置、任务节点、警告和审计记录；重复点击或网络重试时使用同一业务事件，确认幂等结果后再继续。

## 补报与修正

- 普通补报：调度员在位置确认面板补录漏报节点，填写真实驾驶员反馈时间和原因，保存后核对审计记录。
- 管理员修正：仅管理员在发现录入错误后修正位置或时间，并填写修正原因。修正必须保留原始事件和审计痕迹，不能删除或静默覆盖历史。
- 旧事件处理：旧事件可作为历史记录保存，但不得使车辆最新快照回退。发现页面快照异常时停止后续补报并转交管理员核实。

## 异常处置

### 服务区外位置

服务区外位置允许保存，页面会显示警告并生成审计记录。调度员须核对驾驶员反馈、车辆任务和坐标来源；确认误录时走管理员修正，确认真实越界时按运营异常流程登记并持续跟踪。

### 地图不可用

地图、地址搜索或路径能力不可用时，改用经纬度输入或已人工核验的虚拟站点录入。不得将地图加载失败解释为车辆未更新；恢复后复核坐标与任务节点。

### 超过 30 分钟未更新

1. 在位置确认面板或指标中识别超过当前超时阈值的车辆。
2. 联系驾驶员核实车辆状态、最后位置和实际反馈时间。
3. 按实际任务节点补报；无法确认时记录异常原因并升级值班负责人。
4. 恢复上报后复核告警是否清除、审计是否完整以及任务节点是否仍有缺失。

## 每日备份与恢复演练

每日收班后，值班人员应按生产数据库运维规范执行备份、记录备份时间、对象范围、校验结果和保管位置。恢复演练应在隔离环境中从当天备份恢复，核对车辆位置事件、最新快照、任务历史和审计记录的数量与可查询性。

真实生产 PostgreSQL/PostGIS 的备份与恢复演练尚未在本轮验收中执行，状态为“待试点前执行/待关闭”；未完成演练不得标记为通过。

## 指标核对

每班和每日汇总至少核对：上报成功率、录入延迟、超过 30 分钟未更新车辆数、服务区外事件数、管理员修正率、缺失位置任务节点数和查询耗时。异常指标应关联到车辆短号、任务短号、事件数量和审计数量进行人工复核，不在文档中记录个人联系方式。

## 故障联络记录

值班记录使用以下占位字段：`值班负责人：[岗位/工号]`、`技术支持：[支持渠道编号]`、`升级时间：[YYYY-MM-DD HH:mm]`、`事件编号：[编号]`。真实私人联系方式、账号密钥和密码不得写入运行手册、代码库或验收记录。
