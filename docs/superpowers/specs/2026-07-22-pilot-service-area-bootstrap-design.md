# 通渭县试点服务区首建设计

## 目标

在服务区为空且高德行政区边界能力关闭时，系统管理员可以直接在开放瓦片地图上手绘边界，创建“通渭县试点服务区”草稿，并在人工核对后发布启用。该流程承接已完成的规则组首建，为虚拟站点导入和订单服务范围校验提供前置数据。

## 现状与根因

- 后端已经提供 `POST /api/service-areas`，可以创建带边界、运营时段和规则组关联的服务区草稿。
- 资源配置页只实现了已有服务区的边界更新，空库时没有服务区编号，手绘结果无法保存。
- “发布并启用”依赖已保存的草稿边界，因此在零服务区状态下始终不可用。
- 行政区边界导入属于已关闭的高德可选增强能力，不能作为试点首建的必经路径。

## 范围

- 为前端补充服务区创建 API 客户端和创建请求类型。
- 资源配置页同时加载调度规则组，并为零服务区状态提供首建表单。
- 服务区名称默认 `通渭县试点服务区`，运营时段默认 `06:30` 至 `19:00`。
- 默认绑定当前可用的第一套规则组；管理员可以在创建前改选。
- 手绘或粘贴合法 WKT/GeoJSON 边界后创建草稿，成功后自动选中该服务区。
- 创建成功后沿用现有“保存草稿”和“发布并启用”流程。
- 创建、保存边界和发布继续使用后端既有审计逻辑。

本次不修改虚拟站点导入、车辆/驾驶员录入、订单状态机、调度算法和数据库表结构，也不恢复高德 Key 或行政区边界依赖。

## 方案

采用“零状态创建、已有状态编辑”的单组件双模式：

1. 资源页加载服务区、调度规则组、虚拟站点、车辆和驾驶员。
2. 当服务区为空时，电子围栏区域显示创建字段：服务区名称、运营开始时间、运营结束时间和绑定规则组。
3. 地图手绘与边界文本继续复用现有 `ServiceAreaMapEditor`。组件根据是否存在 `serviceArea` 决定提交“创建服务区草稿”或“保存草稿”。
4. 创建模式调用 `POST /api/service-areas`，请求包含名称、边界 WKT、运营时段和规则组编号。
5. 创建成功后把返回值加入资源页服务区列表、设为当前选中项，并同步为虚拟站点默认所属服务区。
6. 页面切换到已有服务区模式，“发布并启用”在存在合法草稿边界时可用。

不采用规则组创建时自动生成空服务区，也不通过 SQL 或数据迁移写入试点服务区。服务区边界必须来自管理员明确提交的手绘或文本输入。

## 接口与数据

复用既有接口：

`POST /api/service-areas`

请求字段：

- `name`：必填，默认“通渭县试点服务区”。
- `boundaryWkt`：必填，闭合 Polygon，业务坐标统一为 GCJ-02。
- `serviceStart`：必填，默认 `06:30:00`。
- `serviceEnd`：必填，默认 `19:00:00`。
- `ruleSetId`：必填，引用已存在的调度规则组。

成功返回 `201 Created` 和服务区草稿视图。权限继续使用 `RESOURCE_MANAGE`；前端发布入口仍只对系统管理员开放。后端负责边界格式、规则组存在性和唯一性校验，前端校验只用于即时反馈。

## 页面状态

### 零服务区

- 显示“新建服务区”及默认字段。
- 没有规则组时禁用创建，并提示先完成调度规则组配置。
- 没有合法边界时禁用创建。
- 高德行政区导入入口不作为必需操作；开放瓦片不可用时仍允许输入 WKT/GeoJSON。

### 创建中

- 禁用创建、刷新和重复提交。
- 保留地图图形、边界文本和业务字段。

### 创建成功

- 自动选中新服务区并显示“服务区草稿已创建，请核对后发布”。
- 虚拟站点新增表单自动带出该服务区。
- 不自动发布，必须由管理员再次确认。

### 创建失败

- 显示后端返回的中文业务错误。
- 保留名称、时段、规则组和边界内容，允许修正后重试。

## 一致性与审计

- 创建草稿和 `SERVICE_AREA_CREATED` 审计记录由后端在同一事务中提交。
- 创建成功前不修改前端服务区列表，也不启用发布按钮。
- 发布继续生成 `SERVICE_AREA_PUBLISHED` 审计记录。
- 订单仅按已发布服务区校验；未发布草稿不能进入试点订单服务范围。

## 测试与验收

- API 回归：创建合法服务区返回 201；无效边界、空名称、无效规则组和无权限请求被拒绝。
- 组件测试：零服务区时显示默认名称、时段和规则组；没有边界时创建按钮不可用。
- 页面测试：提交创建请求的字段正确；成功后自动选中并允许发布；失败后输入不丢失。
- 回归测试：已有服务区的边界保存、发布和虚拟站点管理行为保持不变。
- 本机验收：管理员手绘边界并创建草稿，确认页面显示草稿版本；经公交企业人工核对范围后再执行发布，不以测试多边形替代正式试点边界。

## 分支与交付约束

- 本功能位于 `codex/pilot-service-area-bootstrap`，基于规则组首建提交开发。
- 规则组首建必须先合并；服务区功能随后单独评审和合并。
- 不在本任务中清理、覆盖或伪造现有试点业务数据。
