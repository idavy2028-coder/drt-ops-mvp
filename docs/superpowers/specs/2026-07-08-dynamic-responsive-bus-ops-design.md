# 区域动态响应公交运营管理系统第一版设计

## 1. 产品定位与第一版边界

第一版系统定位为面向公交企业的区域动态响应公交运营管理后台。核心目标不是先做完整乘客端或司机端，而是先把企业内部的实时调度运营闭环跑通。

第一版主链路：

乘客需求录入或接入 -> 虚拟站点匹配 -> 实时调度计算 -> 找到车才接单 -> 自动派发或人工确认 -> 生成车辆任务 -> 执行状态跟踪 -> 异常处理 -> 基础运营指标沉淀。

第一版重点模块：

- 调度工作台：系统核心页面，调度员查看订单、车辆、任务、地图、算法推荐和异常。
- 资源配置：配置服务区域、虚拟站点、车辆、司机、车型、规则。
- 调度算法服务：处理实时匹配、顺路插单、路径评估、综合评分、人工确认判断。
- 订单与任务管理：管理订单状态、车辆任务、执行节点、异常记录。
- 基础运营看板：展示订单量、完成率、等待时间、绕行时间、车辆利用率等基础指标。

第一版暂不重点做：

- 完整乘客端，只保留需求录入和接口接入能力。
- 完整司机端，可先通过后台模拟或简化任务状态上报。
- 完整监管端，只预留数据口径和审计记录。
- 复杂支付、会员、营销、客服体系。
- 大规模机器学习预测，先用规则、启发式评分和可解释调度。

## 2. 核心业务流程与状态闭环

第一版要确保每一笔需求都能被系统清楚地推进到最终结果。

### 2.1 主流程

1. 需求进入
   需求可以来自后台代录、接口接入、未来乘客端。第一版不强依赖真实乘客端，但数据结构按真实乘客需求设计，包括起点、终点、出发时间、人数、联系方式、特殊要求。

2. 虚拟站点匹配
   系统根据乘客原始起终点，在服务区域内匹配可用的虚拟上车点和下车点。匹配失败时，不进入调度，直接标记为不可服务。

3. 实时调度计算
   每个新需求触发一次调度计算。系统先找可用车辆和已有任务，再判断能否顺路插入，最后生成一个或多个候选方案。

4. 接单决策
   如果没有可行方案，订单不确认。如果有低风险最优方案，自动确认并派发。如果方案可行但命中人工规则，进入调度员确认。

5. 任务生成与调整
   确认后的订单会绑定到车辆任务中。任务可能是新建任务，也可能是插入已有任务。任务里要保存站点顺序、预计到达时间、乘客上下车关系。

6. 执行状态跟踪
   第一版可以由后台模拟或简化录入执行状态，包括车辆出发、到达上车点、乘客上车、到达下车点、乘客下车、任务完成。

7. 异常处理
   包括车辆故障、司机不可用、乘客取消、乘客未到、严重延误、临时改派。异常处理后要么重算调度，要么人工关闭，要么转为未完成。

8. 运营沉淀
   每次调度、派发、人工干预、异常处理都留下记录，用于后续看板、复盘、算法评估和监管预留。

### 2.2 状态类型

订单状态：

- 待调度
- 不可服务
- 待人工确认
- 已确认
- 已取消
- 执行中
- 已完成
- 异常关闭

车辆任务状态：

- 待发车
- 已派发
- 执行中
- 暂停
- 已完成
- 已取消
- 异常

调度决策状态：

- 无可行方案
- 自动派发
- 待人工确认
- 人工改派
- 重算成功
- 重算失败

## 3. 调度算法服务边界

调度算法服务是第一版的核心。第一版采用规则约束、路径插入评估、综合评分和可解释结果，不一开始承诺复杂机器学习。

### 3.1 输入

- 新订单：人数、期望出发时间、原始起终点、匹配后的虚拟上下车点。
- 车辆状态：当前位置、载客人数、容量、车型、是否在线、是否可派。
- 司机和任务状态：当前任务、后续站点、预计到达时间、工作时段。
- 区域规则：服务区域、虚拟站点、站点服务半径、最大等待、最大绕行、预约窗口。
- 派发规则：自动派发阈值、人工确认阈值、执行中插单策略。
- 评分权重：等待时间、绕行时间、空驶距离、满载率、准点风险、已有订单影响。

### 3.2 处理步骤

1. 硬约束过滤
   先排除明显不可行方案：车辆不可用、超容量、超区域、虚拟站点不可达、超过最大等待或绕行、司机任务冲突。

2. 候选任务搜索
   在可用车辆和已有任务中找候选对象。优先考虑正在服务区域附近、方向相近、容量可用、时间窗接近的车辆任务。

3. 路径插入评估
   尝试把新订单的上车点和下车点插入已有任务站点序列，评估新增行驶时间、已有乘客绕行变化、预计到站变化。

4. 综合评分
   对可行方案打分。默认偏向乘客体验和服务稳定，即等待更短、对已有乘客影响更小的方案优先。

5. 派发决策
   分为三种结果：无可行方案、自动派发、待人工确认。

6. 解释输出
   每次调度都输出解释信息：候选车辆数量、淘汰原因、最优方案评分、预计等待、预计绕行、是否影响已有订单、为什么自动派发或为什么需要人工确认。

### 3.3 第一版算法边界

- 支持新订单实时插入已有任务。
- 默认只允许同方向顺路插入执行中任务。
- 支持即时出行和未来 2 小时内短时预约。
- 支持后台配置权重和阈值。
- 不做大规模全局最优求解，避免实时响应过慢。
- 不承诺机器学习预测，后续可基于运营数据升级为预测模型。

## 4. 系统模块拆分与功能边界

第一版拆成 6 个核心业务模块。

### 4.1 调度工作台

核心职责：

- 查看实时订单、车辆、任务、异常。
- 展示地图态势：服务区域、虚拟站点、车辆位置、任务路线。
- 查看算法推荐方案和解释。
- 处理待人工确认订单。
- 执行人工改派、取消、重算、异常关闭。

边界：调度工作台负责看和操作，不直接承载算法计算本身。

### 4.2 订单中心

核心职责：

- 管理乘客需求和订单生命周期。
- 保存原始起终点、匹配后的虚拟上下车点、人数、时间窗。
- 管理订单状态：待调度、待确认、已确认、执行中、完成、取消、异常。
- 支持后台代录、接口接入、未来乘客端接入。

边界：订单中心不决定派哪辆车，只保存订单事实和状态。

### 4.3 车辆任务中心

核心职责：

- 管理车辆任务，即一辆车在一段时间内要执行的站点序列。
- 保存车辆、司机、站点顺序、预计到达时间、乘客上下车关系。
- 支持任务新建、插单、重排、暂停、完成、取消。
- 记录任务执行过程。

边界：任务中心保存已形成的执行计划，不负责生成最优方案。

### 4.4 资源与规则配置

核心职责：

- 配置服务区域、虚拟站点、站点服务半径。
- 管理车辆、司机、车型、容量、车辆可用状态。
- 配置预约窗口、最大等待、最大绕行、插单策略、人工确认规则、评分权重。
- 后续兼容通勤班线时，在这里扩展线路、班次、固定站点配置。

边界：配置模块提供规则和资源，不处理实时调度。

### 4.5 调度算法服务

核心职责：

- 接收订单调度请求。
- 搜索候选车辆和任务。
- 做路径插入评估。
- 输出方案评分、派发建议和解释。
- 判断自动派发、人工确认或无可行方案。

边界：算法服务不直接修改订单和任务，建议通过调度编排层确认后再落库，避免算法结果和业务状态混在一起。

### 4.6 运营看板与审计

核心职责：

- 展示基础指标：订单量、完成率、取消率、平均等待、平均绕行、车辆利用率。
- 记录调度决策、人工干预、规则变更、异常处理。
- 为后续监管端、补贴申报、算法评估预留数据基础。

边界：第一版只做基础看板，不做复杂 BI 和预测分析。

## 5. 核心数据对象与关系

第一版核心对象如下。

### 5.1 服务区域 Service Area

表示企业开通动态响应公交服务的地理范围。

关键字段：

- 区域名称
- 区域边界
- 服务时间
- 适用规则组
- 是否启用

关系：

- 一个服务区域包含多个虚拟站点。
- 一个服务区域绑定一套或多套调度规则。

### 5.2 虚拟站点 Virtual Stop

表示系统允许乘客上下车的推荐点。

关键字段：

- 站点名称
- 经纬度
- 服务半径
- 可上车或可下车属性
- 停靠安全说明
- 启用状态

关系：

- 订单会引用上车虚拟站点和下车虚拟站点。
- 车辆任务由多个虚拟站点组成执行序列。

### 5.3 预约需求 / 订单 Ride Order

表示一次乘客出行需求和系统接单结果。

关键字段：

- 原始起点、原始终点
- 匹配后的上车点、下车点
- 乘客人数
- 即时或预约类型
- 期望出发时间
- 预计上车时间、预计到达时间
- 订单状态
- 取消或失败原因

关系：

- 一个订单最终绑定一个车辆任务。
- 一个订单对应一个上车节点和一个下车节点。
- 一个订单可以有多次调度尝试记录。

### 5.4 车辆 Vehicle

表示可用于运营的车辆资源。

关键字段：

- 车牌号
- 车型
- 座位数或容量
- 当前状态
- 当前定位
- 所属车队
- 是否可派

关系：

- 一辆车可以有多个历史车辆任务。
- 同一时间一辆车只能执行一个主任务。

### 5.5 司机 Driver

表示可执行运营任务的驾驶员。

关键字段：

- 姓名
- 联系方式
- 资质状态
- 当前排班
- 当前状态
- 所属车队

关系：

- 一个车辆任务绑定一个司机。
- 司机可用性会影响调度候选集。

### 5.6 车辆任务 Vehicle Task

表示一辆车在一段时间内执行的一串接送计划。

关键字段：

- 车辆
- 司机
- 任务状态
- 计划开始和结束时间
- 站点序列
- 当前执行节点
- 载客变化
- 任务来源：新建、插单、重算、人工改派

关系：

- 一个车辆任务包含多个任务节点。
- 一个车辆任务可以服务多个订单。

### 5.7 任务节点 Task Stop

表示车辆任务中的一个停靠动作。

关键字段：

- 虚拟站点
- 节点类型：上车、下车、途经
- 关联订单
- 计划到达时间
- 实际到达时间
- 节点状态

关系：

- 一个订单至少对应一个上车节点和一个下车节点。
- 节点顺序是路径规划和插单评估的核心。

### 5.8 调度决策 Dispatch Decision

表示算法或人工对某次订单调度的判断过程。

关键字段：

- 关联订单
- 候选车辆或任务数量
- 最优方案
- 评分明细
- 淘汰原因
- 决策结果：自动派发、待人工确认、无可行方案、人工改派
- 决策解释
- 操作人或算法版本

关系：

- 一个订单可以有多条调度决策记录。
- 审计、看板、算法复盘都依赖它。

### 5.9 规则组 Dispatch Rule Set

表示企业配置的一组调度规则。

关键字段：

- 最大等待时间
- 最大绕行时间
- 预约窗口
- 插单策略
- 自动派发阈值
- 人工确认阈值
- 评分权重
- 适用区域、时段、车型

关系：

- 服务区域引用规则组。
- 算法服务每次调度读取规则组。

核心关系原则：

- 订单记录乘客需求与服务承诺。
- 车辆任务记录车辆执行计划。
- 任务节点记录站点级执行过程。
- 调度决策记录为什么这么派。

## 6. 系统架构与数据流

第一版采用业务后台、调度编排层、算法服务的分层架构。后台负责人机操作，编排层负责把订单、任务、规则串起来，算法服务只负责算方案和给解释。

### 6.1 企业管理后台

面向调度员、运营管理员、车队管理员。

包含：

- 调度工作台
- 订单中心
- 车辆任务中心
- 资源配置
- 规则配置
- 基础运营看板
- 审计日志

职责：展示、查询、人工确认、人工改派、异常处理。

### 6.2 业务服务层

负责稳定地管理业务状态。

包含：

- 订单服务
- 车辆任务服务
- 车辆与司机服务
- 区域与虚拟站点服务
- 规则服务
- 审计服务
- 指标统计服务

职责：保存事实，比如订单是什么状态、车辆任务是否执行中、规则当前怎么配置。

### 6.3 调度编排层

这是连接业务和算法的关键层。

职责：

- 接收新订单事件。
- 读取订单、车辆、司机、任务、规则。
- 调用算法服务获取候选方案。
- 判断自动派发、人工确认或拒绝接单。
- 把确认后的结果写回订单和车辆任务。
- 记录调度决策和审计日志。

编排层应独立存在，不让算法服务直接改订单状态。

### 6.4 调度算法服务

职责：

- 虚拟站点匹配。
- 候选车辆和任务搜索。
- 路径插入评估。
- 综合评分。
- 输出方案解释。
- 返回无可行方案、自动派发建议或人工确认建议。

算法服务输出建议和解释，不直接拥有业务最终状态。

### 6.5 外部接入层

后续会接：

- 乘客端订单接口
- 司机端任务接口
- 地图路径服务
- 车辆定位服务
- 支付或票务
- 监管报送

第一版可以先做后台代录和模拟状态，但接口边界要预留。

### 6.6 核心数据流

新需求进入后，先进入订单服务创建待调度订单；调度编排层读取规则和资源，调用算法服务；算法服务返回候选方案和解释；编排层根据派发规则决定自动派发、人工确认或拒绝；自动派发会写入订单、车辆任务和任务节点；人工确认会进入调度工作台；执行过程中车辆任务状态变化继续回写订单和运营指标。

架构原则：

- 订单服务记录需求事实。
- 任务服务记录执行计划。
- 算法服务负责计算建议。
- 调度编排层负责业务决策落地。
- 调度工作台负责人机协同。

未来从实时调度扩展到混合调度时，可以增加批量触发器或定时调度任务，仍然复用同一套调度编排和算法服务。

## 7. 异常处理、审计与验证口径

第一版必须覆盖核心异常，因为动态响应公交的关键能力就在于处理实时变化。

### 7.1 核心异常

- 无可行方案：没有可用车辆、超出服务区域、超出预约窗口、虚拟站点不可达。
- 待人工确认超时：算法给出方案但调度员未及时确认。
- 乘客取消：已确认订单取消，可能影响车辆任务和其他乘客。
- 乘客未到：车辆到达虚拟站点后乘客未上车。
- 车辆故障或离线：已派发任务无法继续执行。
- 司机不可用：司机临时下线、资质异常、排班冲突。
- 严重延误：预计等待或到达时间突破承诺阈值。
- 执行中插单失败：新订单无法插入现有任务，只能新建任务或拒绝。

### 7.2 异常处理原则

- 能自动恢复的，系统自动重算并记录原因。
- 影响已有乘客承诺的，默认进入人工确认。
- 无法恢复的，订单或任务进入异常关闭，并记录原因。
- 所有人工操作必须留痕：谁操作、何时操作、为什么操作、影响哪些订单。
- 异常结果要进入看板指标，不能只停留在日志里。

### 7.3 审计记录

第一版至少记录：

- 订单创建、确认、取消、完成、异常关闭。
- 调度算法每次决策的输入摘要、输出方案、评分和解释。
- 调度员人工确认、改派、取消、重算。
- 规则配置变更。
- 车辆任务状态变化。
- 关键异常事件。

审计记录后续可以支持监管报送、投诉复盘、企业内部责任追踪和算法优化。

### 7.4 第一版验证口径

第一版是否成功，应看这些业务指标是否能跑出来：

- 订单确认率：有多少需求能找到车并成功接单。
- 平均等待时间：乘客从下单到预计上车的时间。
- 平均绕行时间：拼单带来的额外时间。
- 自动派发率：多少订单无需人工确认。
- 人工确认率：多少订单需要调度员介入。
- 任务完成率：已派发任务最终完成比例。
- 异常关闭率：无法完成的订单或任务比例。
- 车辆利用率：车辆运行时间、载客时间、空驶时间的关系。
- 算法解释覆盖率：每次调度是否都有可追溯解释。

## 8. 第一版页面范围与验收标准

第一版按能跑通运营闭环来收页面，不追求大而全。

### 8.1 调度工作台

必须包含：

- 实时订单列表：待调度、待人工确认、已确认、异常。
- 车辆任务列表：待发车、执行中、异常、完成。
- 地图态势：服务区域、虚拟站点、车辆位置、任务路线。
- 算法推荐详情：候选车辆、评分、预计等待、绕行影响、派发建议。
- 人工操作：确认派发、拒绝接单、重算、改派、异常关闭。

### 8.2 资源配置

必须包含：

- 服务区域管理。
- 虚拟站点管理。
- 车辆管理。
- 司机管理。
- 车型和容量配置。

### 8.3 规则配置

必须包含：

- 即时和预约窗口。
- 最大等待时间。
- 最大绕行时间。
- 执行中插单策略。
- 自动派发阈值。
- 人工确认阈值。
- 综合评分权重。

### 8.4 订单中心

必须包含：

- 后台代录需求。
- 订单查询。
- 订单状态流转。
- 订单取消、异常关闭。
- 查看订单调度记录。

第一版不做完整乘客端，但订单数据结构按真实乘客需求设计。

### 8.5 车辆任务中心

必须包含：

- 查看任务站点序列。
- 查看任务关联订单。
- 模拟或录入执行状态。
- 任务暂停、完成、异常关闭。
- 查看任务重算和改派记录。

第一版不做完整司机端，但要能模拟司机执行链路。

### 8.6 基础运营看板

先做基础指标：

- 订单量。
- 订单确认率。
- 自动派发率。
- 人工确认率。
- 平均等待时间。
- 平均绕行时间。
- 任务完成率。
- 异常关闭率。
- 车辆利用率。

### 8.7 审计日志

必须能查：

- 订单状态变化。
- 调度决策记录。
- 人工确认、改派、重算。
- 规则变更。
- 车辆任务状态变化。
- 异常处理记录。

### 8.8 第一版验收标准

第一版至少要跑通这些场景：

- 能配置一个服务区域和一批虚拟站点。
- 能配置车辆、司机和基础调度规则。
- 能录入即时订单和短时预约订单。
- 系统能自动匹配虚拟上下车点。
- 系统能实时计算候选车辆任务。
- 普通订单能自动派发并生成车辆任务。
- 命中阈值的订单能进入人工确认。
- 无可行方案的订单不会被错误接单。
- 已确认任务能被新订单顺路插入。
- 执行中任务能模拟到站、上车、下车、完成。
- 取消、未到、车辆故障、严重延误能进入异常处理。
- 每次调度都有解释记录。
- 基础运营指标能从真实订单和任务状态中统计出来。

### 8.9 测试重点

后续实现时，测试不只测页面能点，还要测业务规则：

- 状态流转测试：订单、任务、调度决策不能乱跳状态。
- 调度规则测试：等待、绕行、容量、区域、预约窗口必须正确限制。
- 插单场景测试：顺路插单、非顺路拒绝、影响已有乘客时转人工。
- 异常场景测试：取消、车辆故障、司机不可用、人工超时。
- 指标口径测试：确认率、完成率、等待时间、绕行时间要可复算。
- 审计测试：关键操作必须留痕。
