# 企业管理端认证与权限体系设计

**日期：** 2026-07-11  
**状态：** 已确认，待实施  
**适用范围：** 区域动态响应公交运营管理系统的企业管理端与 Spring Boot 业务 API

## 1. 目标与范围

为现有企业内部运营 MVP 增加本地账号登录和基于角色的访问控制（RBAC），使不同岗位只能访问和操作其职责范围内的数据与功能。

本期覆盖：

- 本地账号密码登录、刷新令牌、退出登录。
- 用户创建、启停、密码重置、角色分配。
- 四类预置角色及后端 API 权限控制。
- 前端登录、路由保护、菜单和按钮权限控制。
- 用户与授权操作的审计记录。

本期不覆盖：

- 自助注册、自助找回密码、短信或邮件验证码。
- 企业微信、钉钉、LDAP、OIDC 等外部单点登录的实际接入。
- 多租户数据隔离、用户自定义角色或用户自定义权限组合。

系统按单一公交企业部署。用户模型预留组织或企业归属字段，未来多企业扩展时再对业务数据实施租户隔离。

## 2. 认证架构

认证由 Spring Boot 业务 API 统一负责。Vue 管理端只与业务 API 交互；Python 算法服务继续作为内部调度建议服务，不直接向管理端暴露用户登录入口。

### 2.1 令牌与会话

- 登录成功后返回有效期 15 分钟的 JWT 访问令牌。
- 同时签发有效期 7 天的随机刷新令牌；数据库只保存刷新令牌哈希、到期时间、撤销状态和所属用户。
- 前端将访问令牌仅保存在运行时内存中；刷新令牌通过 `HttpOnly`、`Secure`、`SameSite=Lax` Cookie 发送，前端脚本不得读取。
- 刷新请求必须轮换刷新令牌：旧令牌立即撤销，新令牌写入数据库并返回新的访问令牌。
- 退出登录撤销当前刷新令牌。管理员停用用户或重置密码时撤销该用户全部刷新令牌，并递增用户令牌版本。
- 每次受保护 API 请求都验证 JWT 签名、有效期、用户启用状态和令牌版本，使账号停用和密码重置即时生效。

### 2.2 初始管理员

数据库为空时，仅在明确提供启动配置的情况下创建初始系统管理员。不得在代码、迁移脚本或文档中写死默认密码。初始管理员使用临时密码首次登录后必须修改密码。

未来接入 OIDC 或 LDAP 时，保留相同的用户、角色与权限模型，只替换认证身份来源和登录入口。

## 3. 用户、角色与权限模型

新增以下核心数据对象：

- `UserAccount`：用户名、显示名、密码哈希、启用状态、组织预留字段、令牌版本、首次登录改密标记和时间戳。
- `Role`：角色编码、名称、说明。
- `UserRole`：用户与角色关联。
- `RefreshToken`：刷新令牌哈希、用户、到期时间、撤销时间、创建来源。

第一版角色和权限由系统预置，不提供在页面上自由组合权限的能力。授权代码使用稳定的“资源.动作”权限语义，例如：

- `USER_MANAGE`
- `RULE_MANAGE`
- `RESOURCE_MANAGE`
- `ORDER_CREATE`
- `ORDER_READ`
- `DISPATCH_EXECUTE`
- `MANUAL_REVIEW`
- `TASK_EXECUTE`
- `AUDIT_READ`
- `METRICS_READ`

角色边界如下：

| 角色 | 权限范围 |
| --- | --- |
| 系统管理员 | 用户、角色、调度规则和基础配置管理；读取必要的运营信息。 |
| 调度员 | 查看订单与车辆任务；执行调度、人工复核、任务执行和异常处理。 |
| 运营人员 | 维护资源；录入并查看订单；查看车辆任务和运营看板；不得派单或执行任务状态变更。 |
| 审计人员 | 只读查看审计日志、调度决策和运营指标。 |

运营人员拥有订单录入权限，以保持“运营后台录入需求”的既有闭环；调度动作仍只归调度员。

## 4. API 与前端行为

### 4.1 API 边界

公开接口仅包含：

- `POST /api/auth/login`
- `POST /api/auth/refresh`
- `POST /api/auth/logout`
- 健康检查接口

其他业务接口默认要求认证，并按权限拒绝无权请求。认证失败统一返回 401；认证成功但权限不足统一返回 403，响应不得泄露额外业务信息。

系统管理员专用用户管理接口支持创建账号、分配角色、重置密码、启用和停用账号。禁止通过普通业务接口修改用户角色或账号状态。

### 4.2 前端行为

- 未登录访问业务路由时跳转登录页。
- 登录成功后按角色跳转到其首个可用页面。
- 路由守卫、导航菜单和操作按钮根据权限控制可见性，但不能替代 API 鉴权。
- 发生 401 时，前端只尝试一次刷新令牌；刷新失败后清空本地认证状态并回到登录页。
- 发生 403 时保留当前页面并提示无权限，不自动重试。
- 系统管理员可见“用户与权限”页面；其他角色不显示该入口。

## 5. 安全与审计

- 密码使用 BCrypt 哈希保存，登录失败按账号和来源地址限流，防止暴力破解。
- 用户创建、账号启停、密码重置、角色变更、登录失败、刷新令牌异常和授权拒绝均写入审计日志。
- 调用方身份、目标用户、操作结果和必要的失败原因进入审计上下文；审计人员仅有只读权限。
- 用户角色变更后，后续请求立即使用最新授权结果；已停用用户不能继续使用旧访问令牌。

## 6. 验收与测试

后端至少覆盖：

- 密码哈希、登录成功与失败、刷新令牌轮换、退出登录和令牌撤销。
- 账号停用、密码重置后旧令牌即时失效。
- 四类角色对关键 API 的 401 与 403 边界。
- 用户和授权操作写入审计日志。
- 初始管理员仅在显式配置且数据库为空时创建。

前端至少覆盖：

- 登录成功、刷新失败回登录页、无权限提示。
- 按角色显示导航与操作入口。
- 系统管理员的用户管理流程。

端到端流程至少覆盖：

1. 运营人员登录后录入订单，但不能执行派单。
2. 调度员登录后完成派单或人工复核，并执行任务与异常操作。
3. 审计人员只能查看决策、日志和指标。
4. 管理员停用账号后，该账号的后续请求被拒绝。

## 7. 实施边界

本设计将作为下一阶段实施计划的输入。实施时必须保持 Java 业务后台对订单、任务、审计和规则最终状态的所有权；算法服务不承担终端用户认证和角色授权职责。
