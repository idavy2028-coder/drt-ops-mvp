# 车辆任务操作链路专项排查设计

## 1. 背景与目标

2026-07-13 企业内部真实验收中，调度员 `dispatcher01` 创建并人工确认订单后，生成车辆任务 `9081cbff`。在“车辆任务”页选择该 `DISPATCHED` 任务时，页面详情仍显示首条已完成任务；点击“发车”返回 `401`。因此，当前页面无法完成车辆任务状态机。

本专项的目标是先以可重复证据区分两条链路，再以最小改动恢复“选择正确任务并完成动作”的闭环：

1. 选中任务、时间线和按钮状态必须始终来自同一个任务 ID。
2. 任务动作请求必须携带有效访问令牌，并被后端以 `TASK_EXECUTE` 权限受理。
3. 调度员可在真实页面完成“发车、到站、上车、到站、下车、完成”，审计日志可追溯。

## 2. 已确认事实

| 证据 | 结论 |
| --- | --- |
| `TasksPage.vue` 的 `selectedTask` 在选中 ID 不存在时回退到 `tasks[0]` | 选中状态失效时，页面会静默展示首条任务，掩盖上下文错误。 |
| 任务操作通过 `selectedTask.value.id` 发起请求 | 一旦发生回退，可能会对错误任务发起操作。 |
| `DISPATCHER` 包含 `TASK_EXECUTE`，后端 `POST /api/vehicle-tasks/**` 要求该权限 | 角色模型本身不应拒绝调度员；`401` 优先按令牌缺失、失效或刷新失败排查。 |
| 前端通用请求在 `401` 时调用刷新接口并重试一次 | 需要验证首个动作请求和刷新后的重试请求是否都带有正确认证信息。 |
| 真实复验中任务列表读取成功、动作请求返回 `401` | 不能仅用列表加载成功推断动作请求认证正常。 |

## 3. 排查原则

1. 先记录请求事实，再改代码；不能以猜测修改鉴权或放宽权限。
2. 选中任务不允许回退到另一条任务。未找到选中任务时，页面应明确进入“未选择”状态并禁用动作。
3. 不通过删除认证检查、扩大跨域白名单或给调度员新增非必要权限来规避 `401`。
4. 所有修复均先以失败测试或可保存的运行证据锁定，再实施最小改动。

## 4. 需要验证的假设

### A. 任务上下文链路

- A1：页面点击事件未稳定更新 `selectedTaskId`。
- A2：任务列表重新加载后，旧 `selectedTaskId` 不在新数组中，静默回退到首条任务。
- A3：测试只覆盖了 `aria-pressed` 或按钮启用状态，未覆盖时间线、实际请求路径和非首条任务 ID。

### B. 任务动作认证链路

- B1：动作请求发出时 `authStore.accessToken` 为 `null` 或已失效。
- B2：首次 `401` 后刷新令牌失败，或重试请求未携带更新后的 Bearer 令牌。
- B3：本地前端来源、API 基地址或刷新 Cookie 的跨域配置与运行端口不一致。
- B4：后端 JWT 过滤器因 tokenVersion、角色声明或令牌解析异常拒绝请求；这与 RBAC 权限不足应区分记录。

## 5. 修复完成标准

1. 选择非首条 `DISPATCHED` 任务后，该行具有选中标识，时间线显示该任务的两个站点，且“发车”唯一启用。
2. 点击“发车”请求 URL 包含所选任务 ID，并携带有效 Bearer 令牌；后端返回 `200` 和 `IN_PROGRESS`。
3. 后续站点和完成操作按状态逐步解锁，最终任务为 `COMPLETED`。
4. 前端单元测试、认证请求测试、后端权限测试和真实浏览器复验全部通过。
5. 审计日志包含 `TASK_STARTED`、两次 `TASK_STOP_ARRIVED`、`PASSENGER_BOARDED`、`PASSENGER_ALIGHTED`、`TASK_COMPLETED`，验收记录更新为通过。
