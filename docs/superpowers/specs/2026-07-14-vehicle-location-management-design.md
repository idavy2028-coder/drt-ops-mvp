# 车辆位置管理模块设计

## 1. 背景与目标

通渭县区域动态响应公交试点周期为 1 个月，运营时段为 6:30 至 19:00，投入 4 辆车、4 名驾驶员和 2 名调度员。试点期间不建设司机端 App，不接入车载 GPS。驾驶员通过电话或微信向调度员反馈任务节点和车辆位置，调度员在企业管理后台代为更新。

本模块以“位置事件 + 当前快照”为核心：完整保留人工上报历史，同时为调度算法和工作台提供高效的最新位置读取。位置数据用于试点业务流程验证和基础统计，不作为高精度实时轨迹或最终 ETA、准点率的硬性验收依据。

## 2. 范围与边界

### 2.1 本模块负责

- 任务节点操作时同步记录车辆位置。
- 非任务节点的人工补报和管理员修正。
- 车辆最新位置快照和不可变位置历史。
- 调度地图车辆标记、人工节点链和位置过期告警。
- 位置历史查询、导出、审计和运行指标。
- 为未来 `GPS_DEVICE` 来源保留稳定的位置写入服务。

### 2.2 依赖但不在本模块内实现

- 高德地图 JS API、地址搜索、地理编码和地图点击选点。
- 服务区电子围栏编辑与点位范围校验。
- 虚拟站点批量导入、编辑、启停和地图管理。
- 高德驾车路径、距离和预计时间服务。

这些能力通过明确接口向位置模块提供坐标、标准化地址、服务区判断和虚拟站点信息，分别形成独立设计与实施计划。

### 2.3 明确不做

- 连续轨迹采集、轨迹回放和 GPS 在线状态。
- 司机端或乘客端位置上报。
- 修改或删除原始位置事件。
- 以人工节点位置直接证明高精度 ETA 或准点率。

## 3. 总体架构

位置模块分为四个边界清晰的单元：

1. `VehicleLocationRecorder`：统一接收人工上报，校验坐标、时间、关联对象和幂等编号。
2. `VehicleLocationEventRepository`：只追加保存不可变位置事件，提供按车辆、任务和日期查询。
3. `VehicleLocationSnapshotService`：仅在新事件不早于当前快照时更新车辆最新位置。
4. `VehicleLocationQueryService`：向调度地图、历史页、算法候选组装和导出提供专用 DTO。

任务节点动作由 `TaskExecutionService` 调用 `VehicleLocationRecorder`。任务状态、位置事件、车辆快照和审计日志在同一个数据库事务中提交；任何一步失败则全部回滚。

未来 GPS 接入新增位置来源适配器，将设备数据转换为相同的位置命令并标记 `GPS_DEVICE`。核心事件、快照、查询和地图接口保持不变。

## 4. 数据模型

### 4.1 不可变位置事件

新增 `vehicle_location_events` 表：

| 字段 | 约束与含义 |
| --- | --- |
| `id` | UUID 主键 |
| `vehicle_id` | 必填，关联车辆 |
| `vehicle_task_id` | 可空，关联车辆任务 |
| `task_stop_id` | 可空，关联任务节点 |
| `virtual_stop_id` | 可空，关联虚拟站点 |
| `event_type` | `TASK_STARTED`、`PICKUP_ARRIVED`、`PASSENGER_BOARDED`、`DROPOFF_ARRIVED`、`PASSENGER_ALIGHTED`、`TASK_COMPLETED`、`MANUAL_CORRECTION` |
| `source` | 本轮固定为 `MANUAL_DISPATCHER` |
| `location` | PostGIS 点坐标 |
| `longitude`、`latitude` | API 展示和导出使用的数值坐标 |
| `coordinate_system` | 固定为 `GCJ02` |
| `standardized_address` | 高德或人工确认后的标准化地址 |
| `driver_reported_at` | 驾驶员向调度员反馈的业务时间 |
| `recorded_at` | 服务器写入时间 |
| `recorded_by` | 登录调度员或管理员用户编号 |
| `note` | 可空的业务备注 |
| `correction_reason` | 修正事件必填 |
| `corrects_event_id` | 修正事件关联的原事件编号 |
| `idempotency_key` | 客户端本次操作的唯一编号 |
| `snapshot_applied` | 标识该事件是否推进车辆快照 |
| `outside_service_area` | 标识保存时是否位于服务区外 |

数据库为 `vehicle_id + driver_reported_at`、`vehicle_task_id + driver_reported_at`、`recorded_at` 建立索引，并对 `idempotency_key` 建立唯一约束。数据库触发器拒绝对位置事件执行更新和删除；修正必须追加新事件。

### 4.2 车辆最新位置快照

`vehicles` 表保留现有当前位置并增加：

- 最新标准化地址和坐标体系。
- 最新位置来源。
- 最新驾驶员反馈时间和系统录入时间。
- 最新位置事件编号。
- 最新关联任务编号。

快照只接受 `driver_reported_at` 不早于当前快照的有效事件。较晚录入的旧反馈保存为历史，设置 `snapshot_applied=false`，但不回退车辆位置。

### 4.3 坐标规则

试点中的服务区、虚拟站点、订单和车辆位置统一使用高德 `GCJ-02`。数据库同时保存点坐标与 `coordinate_system=GCJ02`，API 不暴露 WKT 字符串。道路距离和 ETA 由高德路径服务计算；PostGIS 主要用于同坐标体系下的围栏判断和候选点空间筛选。

## 5. 命令接口与事务

### 5.1 任务节点操作

保留现有发车、到站、上车、下车和完成接口路径，为每个操作增加必填 `locationReport` 请求体。请求只接受：

- 经度、纬度、标准化地址。
- 驾驶员反馈时间。
- 可选虚拟站点编号和备注。
- 幂等请求编号。

车辆、任务、任务节点、事件类型、位置来源、录入时间和操作人均由后端生成。一次请求按以下顺序在同一事务中执行：

1. 校验权限、任务状态、任务与车辆关系及幂等编号。
2. 校验坐标、反馈时间和关联虚拟站点。
3. 计算服务区内外状态并生成警告。
4. 创建不可变位置事件。
5. 推进任务及任务节点状态。
6. 按时间规则更新车辆快照。
7. 写入业务审计日志。

重复幂等请求返回第一次成功结果，不产生新事件。任务状态冲突、位置保存失败或审计失败均导致事务回滚。

### 5.2 独立接口

- `POST /api/vehicles/{vehicleId}/location-reports`：普通补报或管理员修正。
- `GET /api/vehicles/locations/latest`：批量返回所有车辆最新快照。
- `GET /api/vehicles/{vehicleId}/location-events`：按日期、任务和事件类型查询单车辆历史。
- `GET /api/vehicle-tasks/{taskId}/location-events`：查询任务位置节点链。
- 车辆列表返回专用 DTO，不直接暴露 JPA 实体和 WKT。

普通补报由调度员提交；修正事件仅系统管理员可提交，必须提供 `corrects_event_id` 和修正原因。任何角色均不能修改或删除事件。

## 6. 时间、警告与错误语义

- `driver_reported_at` 不得晚于服务器当前时间。
- 历史补录通过独立补报接口提交，不通过任务节点接口倒填。
- 旧时间事件成功保存，但返回“已保存历史、未更新车辆快照”。
- 服务区外位置成功保存，同时返回 `OUTSIDE_SERVICE_AREA` 警告并写入审计。
- 参数错误、任务与车辆不匹配、非法状态流转和权限不足返回明确中文业务错误。
- 高德服务不可用时允许使用经纬度或虚拟站点提交位置，并返回地图能力降级提示。
- 系统错误只写入服务端日志，不向前端暴露堆栈或数据库细节。

## 7. 后台交互设计

### 7.1 车辆任务页

任务动作按钮先打开位置确认面板。面板默认带出当前任务节点或车辆最新位置，支持地址搜索、地图点击和虚拟站点选择。驾驶员反馈时间必填，系统录入时间由后端生成。

提交成功后统一刷新任务状态、站点时间线和车辆标记。服务区外位置要求调度员再次确认但不阻止保存。提交失败时保留面板输入，事务失败前不得提前移动地图车辆标记。

### 7.2 调度工作台地图

地图展示服务区、虚拟站点、任务路线和车辆最新位置。车辆标记显示车牌、任务状态、最后反馈时间、位置来源和关联任务，并固定标注“人工上报”。

地图每 15 秒轮询最新快照并保留手动刷新。执行中车辆超过 30 分钟没有新位置事件时显示“位置较久未更新”告警，该阈值通过配置项调整。离散位置点可用虚线连接，但必须说明其为人工节点链而非实际行驶轨迹。

### 7.3 位置历史页

历史页支持按车辆、任务、日期和事件类型筛选。时间线同时展示驾驶员反馈时间、系统录入时间、录入延迟、操作人和修正关系。管理员可发起修正；调度员可查询授权范围和提交普通补报。

位置事件可导出为 CSV。导出行为必须写入审计日志，记录操作者、筛选条件、导出时间和记录数量。

## 8. 权限与安全

- 调度员：任务动作位置上报、普通补报、授权范围内查询。
- 系统管理员：全量历史查询、修正事件和导出。
- GPS 设备：未来使用独立设备身份，不复用人员 JWT。
- 位置历史接口和导出接口必须受 RBAC 保护。
- 位置事件不保存驾驶员或乘客的电话、微信等个人通信内容。
- 管理后台继续使用内置账号和 JWT；试点后再评估 OIDC 或 LDAP。

## 9. 运行指标与数据保留

系统采集以下指标：

- 位置上报成功率。
- 驾驶员反馈到系统录入的平均延迟。
- 执行中车辆超过 30 分钟未更新的数量。
- 服务区外位置事件数量。
- 人工修正率。
- 各任务缺失位置节点的数量。
- 高德接口及位置查询的响应时间和失败率。

试点一个月预计不超过约 1 万条位置事件，全部保留，不执行自动清理。PostgreSQL/PostGIS 每日备份；备份恢复演练纳入试点准备任务。

## 10. 测试策略

- 数据库迁移测试：验证 PostGIS 字段、索引、外键、唯一约束和事件不可变触发器。
- 领域测试：验证快照不回退、修正事件保留原记录、事件类型与任务动作一致。
- 事务测试：位置事件、任务、快照和审计任一失败时全部回滚。
- API 测试：验证 RBAC、幂等、时间校验、服务区外警告、补报、修正和历史筛选。
- 前端测试：验证位置确认面板、失败后保留输入、降级录入、轮询和告警展示。
- 端到端测试：使用 4 辆车完成整条任务链，核对地图、任务状态、历史和导出。
- 高德测试：自动化测试使用模拟响应；本机验收使用真实 Key 验证地址、地图和坐标。

## 11. 模块验收条件

1. 每个任务动作恰好产生一条对应位置事件和一条审计记录。
2. 最新快照与最后一个有效位置事件一致，旧事件不会使快照回退。
3. 重复提交返回第一次结果，不产生重复事件。
4. 执行中车辆超过配置阈值后，在一个 15 秒轮询周期内出现告警。
5. 服务区外位置可保存、可追溯并有明显警告。
6. 管理员能够修正错误位置，原事件仍可查询。
7. 地图不可用时，经纬度和虚拟站点方式仍可完成任务操作。
8. 位置历史和导出结果与任务审计链一致，导出行为本身可审计。
9. 约 1 万条试点位置事件可正常查询和导出，并完成每日备份验证。

## 12. 后续演进

试点通过后，司机端 App 或车载 GPS 通过独立适配器提交位置，来源标记为 `GPS_DEVICE`。连续定位、轨迹压缩、设备认证、离线补传和高精度 ETA 另行设计，不改变本模块的位置事件、快照和查询接口。
