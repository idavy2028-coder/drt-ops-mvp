# 人工复核队列设计

## 目标

让调度工作台能够处理算法返回的 `PENDING_MANUAL_REVIEW` 订单，形成“算法建议 - 调度员确认或拒绝 - 订单和任务状态更新”的企业运营闭环。

## 范围

- 新增待人工复核调度决策的只读队列接口。
- 在调度工作台新增人工复核队列面板。
- 支持确认派单和拒绝派单；拒绝必须填写原因。
- 复用现有 `POST /api/dispatch-decisions/{decisionId}/approve` 与 `POST /api/dispatch-decisions/{decisionId}/reject` 接口。

不新增乘客端、驾驶员端、权限体系、批量复核或新的数据库表。

## 接口设计

新增 `GET /api/dispatch-decisions/manual-review`，仅返回决策结果为 `PENDING_MANUAL_REVIEW` 且订单仍处于待人工复核状态的记录。

每个队列项包含：

- `decisionId`：确认或拒绝操作使用的决策 ID。
- `orderId`：关联订单 ID。
- `passengerName`、`passengerCount`、`requestedDepartureAt`：调度员判断所需的订单摘要。
- `bestVehicleId`、`candidateCount`：算法建议摘要。

确认派单调用既有 approve 接口。拒绝派单调用既有 reject 接口，`reason` 为空时由前端阻止提交，后端仍保留现有校验责任。

## 前端交互

调度工作台替换静态算法解释面板的一部分区域，显示“人工复核队列”。队列为空时显示空状态；有记录时逐条展示订单和候选车辆信息。

- 点击“确认派单”后禁用当前条目的操作，调用 approve 接口。
- 点击“拒绝”后显示当前条目的原因输入框和确认拒绝按钮。
- 提交成功后重新加载订单、车辆任务和复核队列。
- 提交失败后保留当前条目与已输入原因，并显示接口错误。

## 数据流

1. 算法返回人工复核，订单进入 `PENDING_MANUAL_REVIEW`，调度决策落库。
2. 工作台加载人工复核队列，以决策 ID 作为操作主键。
3. 调度员确认时，后端创建或确认车辆任务，订单进入可执行状态。
4. 调度员拒绝时，后端记录拒绝原因，订单进入不可服务状态。
5. 工作台刷新运营数据，队列项消失。

## 测试

- 后端：队列接口只返回待人工复核且订单状态匹配的决策；确认和拒绝复用既有服务测试。
- 前端：队列显示、拒绝原因必填、确认或拒绝成功后触发刷新、接口失败保留错误状态。
- 端到端：创建订单后得到人工复核决策，在工作台确认派单并进入车辆任务执行页。

## 非目标

- 不修改算法评分阈值或强制自动派单。
- 不展示完整算法评分明细或候选路径编辑。
- 不处理多调度员并发抢单。
