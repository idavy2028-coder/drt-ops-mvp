# 通渭县试点规则组首建设计

## 目标

在调度规则组为空时，系统管理员可以在后台创建第一套规则组，使规则配置页不再停留在不可保存的空状态，并为后续服务区创建提供可绑定的规则组。

## 范围

- 新增受 `RULE_MANAGE` 权限保护的规则组创建接口。
- 在规则配置页增加“新建规则组”入口、规则组名称输入和创建反馈。
- 将已确认的通渭县试点参数作为新建表单默认值，允许管理员在创建前修改。
- 创建成功后自动选中该规则组，随后沿用现有保存接口继续编辑。

本次不创建服务区、虚拟站点、车辆、驾驶员或订单；这些作为后续独立任务处理。

## 默认参数

| 项目 | 默认值 |
| --- | --- |
| 名称 | 通渭县试点动态调度规则 |
| 最大等候时间 | 5 分钟 |
| 最大绕行时间 | 8 分钟 |
| 预约窗口 | 60 分钟 |
| 自动派单阈值 | 82 |
| 人工复核阈值 | 62 |
| 等候、绕行、稳定性、利用率权重 | 0.35、0.20、0.30、0.15 |
| 执行中插单策略 | `REALTIME_INSERTION`（实时插单） |

## 架构与数据流

1. 管理员在规则配置页点击“新建规则组”，填写名称并确认预填参数。
2. 前端调用 `POST /api/dispatch-rule-sets`，请求体包含名称和完整规则字段。
3. 后端使用既有领域对象创建并持久化规则组，返回创建后的规则组；数据库字段与既有 `dispatch_rule_sets` 表保持一致。
4. 前端将返回值插入列表并设为当前选中项，显示成功提示；之后的“保存规则”继续调用既有 `PUT /api/dispatch-rule-sets/{id}`。

## 接口边界

`POST /api/dispatch-rule-sets`

- 权限：`RULE_MANAGE`。
- 请求：`name` 与现有更新接口的全部规则字段。
- 成功：`201 Created`，返回新规则组。
- 校验：名称必填；数值、阈值和策略沿用现有更新接口约束。
- 失败：返回既有中文业务错误格式；创建失败时前端保留表单内容。

## 交互与错误处理

- 没有规则组时，页面显示明确的空状态和“新建规则组”按钮，不显示可误解为可保存的空表单。
- 创建按钮仅在名称和必填数值有效时可用；提交期间防止重复提交。
- 创建成功后自动进入编辑状态；创建失败时显示中文错误，不清空用户输入。
- “实时插单”和“仅新任务”均为合法策略，保存可用性只取决于是否已有规则组及是否正在提交。

## 测试与验收

- 后端 API 测试：有 `RULE_MANAGE` 权限可创建；无效名称或等待时间被拒绝。
- 前端页面测试：零规则组时显示新建入口；提交默认参数后发出正确请求；成功后页面可保存规则。
- 回归：既有规则组的选择和更新测试继续通过。
- 手工验收：用管理员账号创建“通渭县试点动态调度规则”，页面自动选中规则组，保存按钮可点击。

## 约束

- 不通过 SQL 直接补写规则组，不恢复已清除的演示数据。
- 不改动服务区、虚拟站点、车辆、驾驶员、订单和调度状态机。
- 所有中文界面文案保持 UTF-8；不得以技术状态或静默失败替代业务提示。
