# 任务选择与运营日统计修复设计

## 背景

2026-07-13 企业内部验收中，发现两项影响试点判断的问题：

1. 车辆任务页在存在历史任务和新派发任务时，验收自动化未能将选中任务切换到新任务，导致不能确认页面操作区是否随选择项更新。
2. 运营看板前端以 `Asia/Shanghai` 的日期请求指标，后端却对 UTC 时间直接调用 `toLocalDate()`。因此 UTC `23:00` 至 `23:59` 创建的本地次日订单会被统计到前一天。

第一项的现有前端实现已包含 `selectedTaskId` 更新逻辑，且历史手工验收曾显示选中行高亮。因此先将它作为可复现性问题处理，不能在未复现前假设生产代码存在缺陷。

## 目标

- 任务页能够可靠地选择任意任务；选中任务的高亮、站点时间线和操作按钮必须一致。
- 运营指标的订单和任务均按 `Asia/Shanghai` 运营日归属。
- 使用自动化测试和本地真实流程重新验证完整调度闭环。

## 非目标

- 不调整调度算法、阈值或动态插单策略。
- 不新增驾驶员端、乘客端、监管端或生产试点能力。
- 不修改看板指标定义、接口路径或前端展示格式。

## 设计方案

### 一、任务选择的可复现验证和最小修复

在 `TasksPage` 的页面测试中构造两条任务：首条已完成，第二条为 `DISPATCHED` 且有两个 `PLANNED` 站点。测试点击第二条任务的“选择”按钮后，必须同时验证：

- 第二行的 `aria-pressed` 为 `true`，首行变为 `false`。
- 站点时间线展示第二条任务的站点内容。
- “发车”按钮可用；“到站、上车、下车、完成”仍遵守 `DISPATCHED` 状态规则。

若测试失败，页面使用显式 `selectTask(taskId)` 函数处理选择，并让 `selectedTask` 在 `selectedTaskId` 无效时返回 `null`，不再静默回退到数组首项。首次加载仍明确选择首条任务，刷新后仅当当前选择不存在时才回退到首条任务 ID。

若该测试直接通过，则判定此前现象来自验收自动化环境，不改动任务选择业务代码；只把该测试保留为回归保护，并在验收记录中修正结论。

### 二、上海运营日统一计算

后端 `OperationsMetricsService` 定义唯一运营时区常量：

```java
private static final ZoneId OPERATING_ZONE = ZoneId.of("Asia/Shanghai");
```

订单的 `requestedDepartureAt` 和任务的 `plannedStartAt` 都通过 `atZoneSameInstant(OPERATING_ZONE).toLocalDate()` 转换为运营日。未传日期时，控制器和服务层都使用 `LocalDate.now(OPERATING_ZONE)`。

前端 `DashboardPage` 已使用 `Intl.DateTimeFormat(..., { timeZone: "Asia/Shanghai" })` 生成请求日期，保持不变。该边界使前后端都以同一运营时区解释日期，而 API 仍只传递 ISO 日期字符串。

### 三、测试和验收边界

- 前端页面测试负责验证任务选择与按钮启用状态。
- 后端服务测试负责验证 UTC 日界前后的订单、任务、决策和指标归属。
- 本地真实验收负责验证浏览器页面、调度、任务执行、看板和审计链路。
- 验收记录必须区分“页面通过”“接口通过”和“试点风险”，不能用接口结果替代页面交互结论。

## 验收标准

1. 两条任务的页面测试可证明选择第二条任务会正确切换操作上下文。
2. UTC `2026-07-12T23:17:00Z` 的订单和任务，在查询 `2026-07-13` 时被统计，在查询 `2026-07-12` 时不被统计。
3. 前端类型检查、相关单测、后端相关测试和全量验证通过。
4. 重走本地真实流程后，任务页面能执行新任务，看板显示当日订单量和完成率，审计日志完整。
